# リサーチと設計判断

---
**目的**: 設計に影響する調査結果・検討過程・判断理由を記録する。

**使い方**:
- ディスカバリ工程の調査ログを残す
- `design.md` に載せきれない比較や根拠を整理する
- 将来の監査・再利用の参照点にする
---

## サマリー
- **機能**: agent-core
- **ディスカバリ種別**: 既存機能の拡張
- **主要な発見**:
  - LangGraph の StateGraph はビルダであり compile が必須、ノードは状態の部分更新を返す設計である。
  - Typer は Annotated を用いた引数定義で、型・ヘルプ・検証を一体化できる。
  - 外部情報の参照は ToS と許可された API のみに限定し、クローリングは禁止とする必要がある。

## 調査ログ

### LangGraph StateGraph とワークフロー構成
- **背景**: Profile/Job/Evaluate のワークフローを共通パターンで設計するため。
- **参照元**: LangGraph 公式ドキュメント（Overview / Graph API / Reference）
- **発見事項**:
  - StateGraph はノードとエッジを明示的に定義し、compile 後に実行可能なグラフとなる。
  - 状態は TypedDict や dataclass で定義でき、ノードは部分更新を返す。
- **設計への影響**: 各エージェントは LangGraph を共通オーケストレーション基盤として採用し、状態モデルとノード責務を分離する。

### Typer の CLI パターン
- **背景**: CLI 入力検証とオプション設計を統一するため。
- **参照元**: Typer 公式ドキュメント
- **発見事項**:
  - Annotated を用いた引数宣言で型情報とヘルプを一体化できる。
  - Option の制約（存在チェック、readable など）を CLI 層で担保できる。
- **設計への影響**: CLI 入力の検証は Typer に寄せ、セッション構築前に前提条件を満たす。

### 外部情報源の利用制約
- **背景**: 求人・会社情報の補完に外部情報源を使う場合の制約整理。
- **参照元**: 一般的な求人サイト ToS 例、プロジェクト方針
- **発見事項**:
  - クローリング/スクレイピングは禁止とする方針が妥当。
  - ユーザーが取得した入力や、明示的に許可された API 以外は利用不可。
- **設計への影響**: Job Agent の補完は「許可された情報源のみ」に限定し、取得不可時は警告で継続する。

## アーキテクチャパターン評価

| 選択肢 | 説明 | 強み | リスク / 制約 | 備考 |
|--------|------|------|---------------|------|
| レイヤード・ワークフロー | CLI → Session → Workflow → Domain Services → Storage | 既存パターンと一致、責務分離が明確 | 共通処理の設計が必要 | 既存実装と整合 |
| ヘキサゴナル | コアドメイン + アダプタ分離 | テスト容易、外部依存を隔離 | 設計負荷が増加 | 将来拡張に有効 |
| モノリス型モジュール | 1 パッケージに集約 | 実装は簡便 | 境界が曖昧で拡張に不利 | MVP 向きだが長期不利 |

## 設計判断

### 判断: 共通オーケストレーションの採用
- **背景**: Profile/Job/Evaluate を一貫した実行方式で扱い、テスト容易性を確保したい。
- **検討案**:
  1. 各エージェントごとに独自フロー
  2. LangGraph に統一
- **採用案**: LangGraph を共通のワークフロー基盤として使用し、状態とノードを分離する。
- **理由**: 既存実装・ステアリングと整合し、拡張余地が高い。
- **トレードオフ**: すべてのエージェントで状態モデルを定義する必要がある。
- **フォローアップ**: ノード境界と状態更新ルールを実装時に検証する。

### 判断: 会社情報の補完は Job Agent のオプションとする
- **背景**: 会社情報が不足する求人入力に対応する必要があるが、別エージェント化は過剰。
- **検討案**:
  1. Company Research Agent を新設
  2. Job Agent 内で補完を任意実行
- **採用案**: Job Agent 内で許可済み情報源に限定した補完オプションを提供。
- **理由**: 機能の重複を避け、責務の増大を最小化する。
- **トレードオフ**: 会社リサーチの汎用性は限定的。
- **フォローアップ**: 将来必要であれば独立エージェントとして分離。

### 判断: 低品質入力はドラフトとして保存
- **背景**: 欠損入力が多い場合でも中断せず成果物を残したい。
- **検討案**:
  1. 必須項目欠損でエラー終了
  2. 欠損を記録したドラフト保存
- **採用案**: ドラフト保存を採用し、欠損リストを明示する。
- **理由**: 単一ユーザー運用でも作業継続性を確保できる。
- **トレードオフ**: 後続で欠損を解消する運用が必要。
- **フォローアップ**: 評価フェーズで欠損時のエラーを明示する。

## リスクと対策
- 外部ソースの利用条件変更 — 許可済み API のみ使用し、取得不可時は警告で継続。
- LLM 出力のばらつき — 入力プロンプトに出所と時刻を明示し、評価結果を再現可能にする。
- 入力欠損による評価失敗 — 欠損リストを明示し、評価時は阻害要因を返す。
